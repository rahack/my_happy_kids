# Happy Kids — мотивационный бот для детей на каникулах

Прототип Telegram Mini App: админ (родитель) ведёт список ежедневных задач и наград для детей, отмечает выполнение и вручает награды по заполнению шкалы прогресса.

## Стек
- Node.js + Express
- SQLite (better-sqlite3)
- Telegraf (Telegram bot)
- Vanilla JS Mini App (статика из `public/`)

## Структура
```
.
├── server.js        # Express + сессии + REST API + автозапуск туннеля
├── bot.js           # Telegram-бот, кнопка запуска Mini App
├── tunnel.js        # Туннель с авто-fallback: cloudflared → serveo.net → localtunnel
├── db.js            # SQLite, схема (users / kids / tasks / rewards / invites / memberships) + миграции
├── public/          # Mini App: index.html, app.js, style.css
├── tools/
│   └── cloudflared.exe  # бинарь cloudflared (Windows AMD64)
└── .env             # TELEGRAM_TOKEN, WEBAPP_URL?, SESSION_SECRET?
```

## Запуск
1. Зависимости установлены (`npm install` если нужно повторить).
2. `.env`:
   ```
   TELEGRAM_TOKEN=<токен от BotFather>
   SESSION_SECRET=<любая строка>     # опционально
   PORT=3000                         # опционально
   # WEBAPP_URL=https://...          # опционально, если хотите свой URL
   # NO_TUNNEL=1                     # опционально, отключает автозапуск туннеля
   ```
3. Старт:
   ```
   npm start
   ```
   `server.js` автоматически:
   - попробует `tools/cloudflared.exe` (quick tunnel, без регистрации);
   - при сбое cloudflared — **serveo.net** (SSH, без установки);
   - при сбое serveo — **localtunnel** (`*.loca.lt`);
   - URL первого успешного подставит как `WEBAPP_URL` для бота.

   В консоли увидите строки `[tunnel] public URL ready: ...` и `[bot] @<...> ready`.
4. Открыть Mini App:
   - В Telegram: написать боту `/start` → нажать «Открыть».
   - Или в браузере: <http://localhost:3000> (туннель не нужен для локального доступа, но без Telegram-клиента не будет кнопки «Войти через Telegram»).
5. Войти.
   - **Родитель (админ):** через кнопку **«Войти через Telegram»** в Mini App. При первом входе сервер автоматически создаёт под этого Telegram-пользователя новый аккаунт-«семью» (свои дети/задачи/награды). Пароль аккаунта рандомный — пользователь его не вводит, всегда заходит через Telegram. Внутри Settings можно установить **PIN-код** (минимум 4 цифры) — он будет нужен, чтобы перейти из режима «Просмотр» в режим редактирования.
   - **Валидатор:** заводится админом в его настройках (логин + пароль). После этого валидатор заходит обычной формой логин/пароль в нижней части экрана — без Telegram-привязки.

### Туннелирование
Автоматический fallback по цепочке (каждый следующий пробуется только при отказе предыдущего):
1. **cloudflared** — quick tunnel, без регистрации. Бинарь `tools/cloudflared.exe`.
2. **serveo.net** — SSH-туннель, не требует бинарей. Нужен открытый исходящий TCP:22.
3. **localtunnel** — npm-пакет (`localtunnel`), URL вида `*.loca.lt`. Наименее надёжный.

- URL **меняется при каждом перезапуске** — это особенность quick tunnel.
- Чтобы зафиксировать URL: создайте named tunnel в кабинете Cloudflare и пропишите его в `WEBAPP_URL` (тогда автозапуск пропустится).
- Кастомный путь к cloudflared: `CLOUDFLARED_PATH=C:\path\to\cloudflared.exe`.
- Отключить туннелирование: `NO_TUNNEL=1` (только локальный доступ).

## Сценарий

### Настройка (режим «Родитель 🔓»)
1. Войти через Telegram (Mini App создаст для вас семью автоматически).
2. Зайти в «Настройки» → задать **PIN-код родителя** (от 4 цифр). До этого момента переключение в режим «Родитель» происходит мгновенно, без подтверждения; после установки PIN — потребует ввод PIN.
3. Переключиться в режим **«Родитель 🔓»** (кнопка справа вверху).
4. Создать профиль ребёнка (имя, возраст, пол, опционально фото).
5. Открыть профиль → выбрать день (полоса под шкалой) → добавить задачи → назначить награду.
6. (Опционально) В «Настройках» завести валидатора (логин/пароль) — или сгенерировать invite-ссылку, чтобы пригласить валидатора через Telegram (см. ниже).
7. Переключиться в режим **«Просмотр 👀»** и передать устройство ребёнку.

### Выполнение (режим «Просмотр 👀»)
1. Ребёнок отмечает выполненные задачи галочкой. Рядом с задачей появляется оранжевый бейдж **«На проверке»** — задача попала в очередь к валидатору.
2. Валидатор (см. ниже) подтверждает галочку → задача засчитывается, шкала заполняется.
3. Пока не 100% — на месте награды 🎁 «Сюрприз!». Название не светится.
4. На 100% появляется кнопка **«Открыть награду»**.
5. Родитель нажимает → форма ввода **пароля родителя** → «Сюрприз готов!» (название всё ещё скрыто) + кнопка «Вручить награду».
6. Клик по **«Вручить награду»** → 🏆 + название награды наконец-то раскрывается. Ребёнок видит свой приз.

### Подтверждение галочек (режим «Валидатор ✅»)
1. Залогиниться валидатором (логин/пароль, выданный родителем) **или** будучи родителем, нажать «Валидатор ✅» в шапке и ввести логин/пароль валидатора своей семьи.
2. Открывается общий список заданий на проверке (контекста текущей семьи), сгруппированный по детям.
3. Для каждой галочки доступны **«Подтвердить»** (задание идёт в зачёт) и **«Отклонить»** (галочка снимается, ребёнок может попробовать снова).
4. Пока валидатор не подтвердил — задача в прогрессе не считается, на 100% шкала не выходит, награду открыть нельзя.

### Invite-ссылки (пригласить валидатора через Telegram)
1. В режиме «Родитель» в «Настройках» нажать **«Создать приглашение»** — сервер сгенерирует постоянную multi-use ссылку вида `https://t.me/<bot>?start=inv_<token>`.
2. Скинуть ссылку человеку. По клику открывается чат с ботом → inline-кнопка «Принять приглашение» → Mini App открывается с `?invite=<token>`.
3. Принявший становится валидатором в вашей семье (через таблицу `memberships`). Только пользователи с собственной TG-привязкой (т.е. кто хоть раз входил через Telegram сам) могут быть гостевыми валидаторами в чужих семьях.
4. Приглашение действует до удаления (admin может отозвать через **«Удалить»**). Один токен можно redeem-ить много раз — это нормально.
5. У такого «гостящего» админа появляется **семейный switcher** (`<select>`) в шапке: можно переключаться между своей семьёй и семьями, куда его пригласили.

### Просмотр других дней
Полоса дат под шкалой прогресса:
- **Прошлое** — read-only архив с маркерами выполнения и историей наград.
- **Сегодня** — обычный поток.
- **Будущее** — для ребёнка скрыто, родитель может планировать задания и награду заранее.

### Защитные механики
- Снятие галочки на сервере автоматически сбрасывает `claimed=0` → ребёнок не «зафиксирует» награду навсегда снятием+постановкой.
- Изменение названия награды в режиме родителя тоже сбрасывает `claimed=0` — это новая награда, надо заслужить.
- Раскрытие награды требует пароль каждый раз при заходе в профиль (состояние не персистится).
- Переключение из «Просмотр» в «Родитель» требует **PIN-кода родителя** (отдельная сущность от пароля аккаунта; устанавливается в «Настройках»). До установки PIN переключение мгновенное — это осознанное решение для быстрого старта.
- Все данные одного родителя изолированы от других: каждая запись в БД (`kids`/`tasks`/`rewards`) тегирована `owner_id`, запросы фильтруются через `session.contextParentId`.
- Telegram `initData` валидируется HMAC-проверкой (`verifyTelegramInitData`) — нельзя «подделать» свой Telegram id с фронта.

## Три режима интерфейса

Кнопки переключения справа вверху, текущий выбор сохраняется в `localStorage`.

| | Просмотр (по умолчанию) | Родитель | Валидатор |
|---|---|---|---|
| Отметка задач (галочки на сегодня) | ✅ → «На проверке» | ✅ → «На проверке» | — |
| Подтверждение / отклонение галочек | — | — | ✅ |
| Шкала прогресса + календарь дней | ✅ | ✅ | — (только список pending) |
| Просмотр прошедших дней (архив) | ✅ | ✅ | — |
| Раскрытие награды (с паролем + кнопкой «Вручить») | ✅ | — (нет кнопки, чтобы сюрприз показал ребёнок) | — |
| Видеть название награды до вручения | ❌ | ✅ (планирование) | — |
| Планирование на будущие даты | ❌ | ✅ | — |
| Создание/редактирование/удаление детей и фото | ❌ | ✅ | — |
| Добавление/удаление задач | ❌ | ✅ (сегодня + будущее) | — |
| Назначение/изменение награды | ❌ | ✅ | — |
| Статистика и история | ❌ | ✅ | — |
| Настройки, выход | ❌ | ✅ | ✅ (выход) |

Переключатель — UI-удобство, не серверная защита. Но переходы «Просмотр → Родитель» и «Просмотр → Валидатор» **требуют подтверждения**: первый — PIN-код родителя (`/api/verify-pin`), второй — логин/пароль валидатора текущей семьи (`/api/verify-validator`). Это барьер от случайного нажатия ребёнком.

**Важно:** любая галочка (даже поставленная родителем) попадает в очередь к валидатору и не засчитывается, пока тот её не подтвердил. То есть «100% сделано» = «все задачи дня approved валидатором».

## API (для разработки)

### Авторизация
- `POST /api/login {username, password}` — обычный логин по БД. Роль (`admin`/`validator`) определяется по записи в `users`.
- `POST /api/tg-auth {initData}` — Telegram-логин. Валидирует initData по HMAC. При первом входе создаёт нового админа автоматически. Ответ: `{ ok, action: 'login' | 'bound' | 'registered' | 'conflict', username, role }`.
- `POST /api/logout` / `GET /api/me` — возвращает `{ authenticated, username, role, tg_linked, has_pin, context: { parent_id, parent_username, role, is_self }, can_switch_context }`.
- `GET /api/has-kids` — публичная проба для UX логин-формы.

### PIN и подтверждения
- `POST /api/admin-pin {pin, oldPin?}` — установить/изменить PIN родителя (минимум 4 цифры). Если PIN уже установлен — нужен `oldPin`.
- `DELETE /api/admin-pin {pin}` — удалить PIN.
- `POST /api/verify-pin {pin}` — проверить PIN (для view→admin переключения и раскрытия награды). Только для admin-контекста.
- `POST /api/verify-admin {username, password}` — повторная проверка пароля текущего админа без смены сессии.
- `POST /api/verify-validator {username, password}` — проверка кредов валидатора, принадлежащего текущему контексту.
- `POST /api/change-password {oldPassword, newPassword}` — смена пароля текущего юзера (любая роль).

### Multi-family
- `GET /api/my-families` — список контекстов, в которых пользователь может работать (своя семья + семьи по invite).
- `POST /api/switch-context {parent_id}` — переключить активный контекст. Только в рамках доступных семей.
- `GET /api/invites` — список приглашений текущего админа (`{ id, token, created_at, url }`).
- `POST /api/invites` — создать новый invite. Возвращает `{ id, token, url }` (URL вида `https://t.me/<bot>?start=inv_<token>`).
- `DELETE /api/invites/:id` — отозвать invite.
- `POST /api/invites/redeem {token}` — принять приглашение. Текущий юзер становится валидатором в семье admin'а. Только для пользователей с primary-role `admin` (TG-зарегистрированных).

### Валидаторы (CRUD, только admin своей семьи)
- `GET /api/validators` — список локальных валидаторов текущей семьи.
- `POST /api/validators {username, password}` — создать нового валидатора.
- `DELETE /api/validators/:id` — удалить локального валидатора.
- `POST /api/validators/:id/password {password}` — сменить пароль валидатору.
- `GET /api/members` — объединённый список: локальные валидаторы (`type:'local'`) + TG-гости по invite (`type:'tg_member'`). Используется страницей «Валидаторы» в Настройках.
- `DELETE /api/members/:id` — отозвать доступ TG-гостя (удаляет строку из `memberships`, аккаунт не трогает).

### Данные (admin = active context)
- `GET /api/kids` / `POST /api/kids {name, age, gender, photo?}` / `PUT /api/kids/:id {name, age, gender, photo?}` / `DELETE /api/kids/:id` — только в admin-контексте.
- `GET /api/kids/:id` — профиль: `today`, `history`, `stats`, `calendar` (маркеры активности для полосы дат).
- `GET /api/kids/:id/day/:date` — задачи и награда за произвольную дату (используется календарём).
- `POST /api/kids/:id/tasks {date?, title}` (admin) / `POST /api/tasks/:id/toggle` (любая авторизация) / `DELETE /api/tasks/:id` (admin)
  - `toggle` — машина состояний: `open → pending → open`, `approved → open`. Прямо в `approved` через `toggle` попасть нельзя — только через approve валидатора.
  - При снятии галочки автоматически сбрасывается `claimed=0` у награды дня, если после этого задачи не все выполнены.
- `POST /api/tasks/:id/approve` — `pending → approved`, ставит `completed_at`.
- `POST /api/tasks/:id/reject` — `pending → open`.
- `GET /api/pending-tasks` — общий список заданий, ждущих подтверждения, с join'ом по детям (имя + фото) в рамках текущей семьи.
- `POST /api/kids/:id/reward {date?, title}` (admin; upsert; смена title сбрасывает `claimed=0`).
- `POST /api/rewards/:id/claim` (admin; проверяет, что все задачи дня approved).

Все запросы к данным фильтруются по `owner_id = session.contextParentId` — каждая семья видит только свои записи. Фото в `kids.photo` — base64 data URL; клиент даунскейлит до 256×256 (JPEG, q≈0.85). Лимит JSON-body поднят до 2МБ.

## Известные ограничения и нюансы
- Telegram WebApp `initData` валидируется HMAC-проверкой по `TELEGRAM_TOKEN`. Без правильного токена `/api/tg-auth` вернёт 503/401 и регистрация админов через TG не будет работать.
- Multi-tenancy: каждый Telegram-пользователь = свой аккаунт-«семья» (`users.role='admin'`, `parent_id IS NULL`). Изоляция данных через `owner_id` на data-таблицах.
- БД-файл `data.db` создаётся автоматически при первом запуске. **Старые `data.db` (с одной single-row `admin`) несовместимы со схемой `users` — пересоздавайте `data.db` при апгрейде.**
- Quick tunnel URL **меняется при каждом перезапуске**. Бот автоматически переустанавливает Menu Button с новым URL.
- На фронте стоит `Cache-Control: no-store` + cache-buster в URL — Telegram очень любит кешировать Mini App'ы. Если после рестарта в Telegram видите старое — закройте/откройте чат с ботом или Reload Page внутри Mini App.

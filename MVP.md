# MVP

## Исходное ТЗ

Нужно создать телеграмм приложение. Токен для бота указан в `.env`.
Приложение для мотивации детей на летних каникулах.

1. Каждодневный список задач и каждодневная награда за выполнение.
2. Есть всего один пользователь с полными правами, который администрирует список задач по дням и награды. Его можно создать пока с логином `admin` и паролем `admin`. Также должна быть возможность менять пароль.
3. Администратор умеет создавать профили детей.
4. Назначать список задач на день и награду в конце выполнения всех задач на день каждому ребенку.
5. При создании профиля ребенка необходимо ввести имя, возраст, пол.
6. Каждое задание имеет название.
7. Каждая награда имеет название.
8. Администратор открывает профиль ребенка и видит задания прошлых дней и статистику выполнения и видит задания на текущий день.
9. Когда ребенок выполнил задание, администратор это отмечает.
10. Во время выполнения заданий появляется шкала.
11. После выполнения каждого задания шкала заполняется.
12. После заполнения на 100% появляется награда в профиле ребенка.

## Статус: реализовано

| # | Пункт | Где |
|---|---|---|
| 1 | Ежедневный список задач и награда | `tasks` и `rewards` с `date` по дням |
| 2 | Один админ `admin/admin`, смена пароля | `db.js` сидинг, `/api/change-password` |
| 3 | Создание профилей детей | `/api/kids` POST |
| 4 | Назначение задач и награды на день для ребёнка | `/api/kids/:id/tasks`, `/api/kids/:id/reward` |
| 5 | Имя, возраст, пол при создании | форма в `public/app.js` (renderKidsList) |
| 6, 7 | Название у задачи и награды | поля `title` в таблицах |
| 8 | Прошлые дни + статистика + текущий день | `getKidProfile` в `server.js` |
| 9 | Админ отмечает выполнение | `POST /api/tasks/:id/toggle` |
| 10, 11 | Шкала прогресса | `.progress-bar` в `style.css`, `pct = done / total` |
| 12 | Награда на 100% | `renderRewardSection` в `public/app.js` |

## Решения по архитектуре

- **Формат:** Telegram Mini App (WebApp), а не «голый» бот. Бот выступает лаунчером для админ-панели через Menu Button.
- **Хранилище:** SQLite (`better-sqlite3`). Один файл `data.db`, журналирование WAL.
- **Аудитория:** Только админ (родитель). У детей нет отдельного входа в бота — родитель сидит с ребёнком и отмечает галочки.
- **Туннелирование:** автозапуск `cloudflared` quick tunnel для HTTPS-URL (обязателен для Telegram Mini App).
- **Стек фронта:** vanilla JS без сборщиков — минимум зависимостей, мгновенная итерация.

## Расширения сверх ТЗ (по итогам обсуждения)

1. **Два режима интерфейса** (`localStorage.mode`):
   - **«Просмотр»** (по умолчанию) — родитель + ребёнок: галочки, шкала, награда «через пароль».
   - **«Родитель»** — управление профилями, задачами, наградами, настройки.
   - Переключение `view → admin` требует пароля (модалка `renderModeAuthModal`, повторная проверка `/api/verify-admin`). Назад — без пароля.

2. **Просмотр любого дня через календарь.** Под шкалой прогресса — горизонтальная полоса дат (14 дней вокруг выбранного, навигация ±неделя). Точки на ячейках — маркеры активности (есть задания / награда получена). Логика рендера зависит от типа дня:
   - **Прошлое** — read-only архив (включая admin): чекбоксы заморожены в виде ✓/○, награда показана открыто если была получена; иначе с пометкой «Не получена».
   - **Сегодня** — основной поток (галочки, разблокировка награды по паролю).
   - **Будущее** — для view скрыто; admin может планировать задания и награду.
   - Бэк: `GET /api/kids/:id/day/:date` + поле `calendar` в профиле для маркеров.

3. **Награда — «сюрприз» только в момент вручения** (view-режим):
   - Пока не все задачи — 🎁 «Сюрприз!», название скрыто.
   - На 100% — «Открыть награду» → пароль → «Сюрприз готов!» (название всё ещё скрыто) → «Вручить награду» → 🏆 + название.
   - В режиме родителя название видно всегда (планирование); кнопки «Вручить» нет — вручение делается в режиме просмотра, чтобы ребёнок увидел сюрприз.

4. **Серверный инвариант `claimed ⇒ 100%`.** `POST /api/tasks/:id/toggle` сам сбрасывает `claimed=0`, если после переключения задача оказалась < 100%. То есть награда «расвручается» не визуально, а в БД — позволяет повторно заработать её и не путаться в данных.

5. **Профиль ребёнка: редактирование + удаление + фото.** `PUT /api/kids/:id`; фото — base64 data URL, на клиенте даунскейл до 256×256 (JPEG, q=0.85), хранится прямо в колонке `kids.photo`. `express.json({ limit: '2mb' })`.

6. **Лёгкие миграции БД.** Хелпер `ensureColumn` в `db.js` через `PRAGMA table_info` добавляет nullable-колонки без потери данных (так была добавлена `kids.photo`).

7. **Анти-кеш Mini App + тёмная тема Telegram.** `Cache-Control: no-store` + cache-buster в URL Menu Button. Хардкод цветов текста на светлых плашках, чтобы не было «белый на белом».

8. **Роль Валидатора + двухступенчатая авторизация.**
   - Захардкоженный второй пользователь `validator/12345` в `server.js` (отдельный от admin), сессия хранит `role`.
   - В таблице `tasks` добавлена колонка `pending` (миграция через `ensureColumn`). Любая галочка идёт сначала в `pending`; в зачёт прогресса/награды попадает только после `POST /api/tasks/:id/approve` (есть и `reject`). Прямой `toggle` к `approved` запрещён — это инвариант machine state.
   - В UI режима три кнопки: «Просмотр 👀» / «Родитель 🔓» / «Валидатор ✅». Validator-режим показывает страницу `pending` со всеми ждущими подтверждения задачами, сгруппированными по детям.
   - В режиме просмотра рядом с галочкой висит оранжевый бейдж «На проверке».
   - Логин-форма двухступенчатая: сначала выбор роли, потом сами поля. Публичный `GET /api/has-kids` позволяет пропустить выбор на свежей установке (в БД нет детей → сразу форма родителя).

9. **UX-мелочи по итогам ревью.**
   - Шапка календарной шкалы показывает полную дату выбранного дня + текущее время (тикает раз в секунду). Дубль с датой над прогресс-баром убран.
   - Форма «Добавить ребёнка» в режиме родителя свёрнута за кнопку — раньше форма висела сверху всегда.
   - При выходе из режима Родителя сбрасываются открытые админские состояния (форма редактирования профиля, форма добавления).

## Известные ограничения прототипа

- Telegram WebApp `initData` не валидируется на сервере — авторизация по логину/паролю и сессии, как было в ТЗ.
- Переключатель режима «Просмотр»/«Родитель»/«Валидатор» — это UI-удобство, а не строгая серверная защита. На бэке роль есть и часть эндпоинтов гейтится (`requireAdmin` для CRUD), но `approve`/`reject` доступны любой авторизованной сессии (compromise: не хранить «validator-unlocked» в сессии). Для защиты от случайного нажатия ребёнком — достаточно; для защиты от мотивированного — нет.
- Пароль валидатора захардкожен в `server.js` (`VALIDATOR_USER` / `VALIDATOR_PASS`). Менять — правкой кода и рестартом, не через API.
- Quick tunnel URL меняется при каждом перезапуске сервера (для стабильного URL нужен named tunnel в Cloudflare).
- Миграции БД — только лёгкие через `ensureColumn` (добавление nullable-колонок). Несовместимых изменений нет, тестов и CI тоже нет.
